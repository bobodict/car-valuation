# 简洁估值流程设计

## 1. 背景

当前前端以研究控制台为首页，默认先展示模型审计、评估指标和误差分析，再通过侧边栏进入估值。这个结构适合反复检查模型的技术用户，但第一次打开页面的人无法立即判断下一步应该做什么。

本次设计将首页重心调整为“完成一次车辆估值”，同时保留模型证据、历史记录和解释助手。设计参考 Apple 产品常见的克制层级：主任务先完成，复杂信息按需展开；不复制品牌组件或视觉资产。

## 2. 目标与非目标

### 目标

- 用户进入首页后可以立即开始估值，而不是先理解研究控制台结构。
- 将当前 15 个输入字段按用户任务分为 4 步，减少单屏认知负担。
- 每一步独立校验，返回修改不丢失已填信息。
- 结果页优先展示价格、参考区间、质量状态和结果边界。
- 通过“查看估值依据”渐进展示模型版本、指标、数据来源和限制。
- 保持现有 API、请求字段、模型质量门禁和研究数据结构不变。
- 满足键盘操作、语义标签、可读对比度、窄屏布局和 reduced-motion 要求。

### 非目标

- 不修改后端接口、模型训练、模型卡字段或数据库结构。
- 不引入新的 UI 框架、图标库或状态管理库。
- 不删除研究总览、历史记录、解释助手和独立测试信息，只调整默认入口和展示层级。
- 不把模型输出包装成统计置信区间或交易承诺。

## 3. 目标用户与核心任务

主要用户是希望快速获得二手车参考价格的普通用户，以及需要反复比较估值结果的业务用户。技术用户仍然需要查看模型证据，但这不是第一次打开页面时的首要任务。

核心路径：

```text
打开首页 -> 开始估值 -> 填写 4 个步骤 -> 检查输入 -> 提交
       -> 先看价格与区间 -> 按需展开估值依据 -> 修改或查看历史
```

## 4. 信息架构

### 顶部导航

- 左侧：车辆估值品牌标识。
- 右侧：模型说明、历史记录、解释助手、API 状态。
- 不再使用占据固定宽度的编号侧边栏作为主要导航。
- 研究总览作为“模型说明”的详情页或详情区块进入，不作为默认首页。

### 首页

首页默认显示估值向导。标题直接说明任务，例如“这辆车值多少？”，副标题说明输出是参考价格而非交易承诺。

首页只保留一个明确的主行动按钮体系：继续、返回、开始估值。重置属于次级操作，并且不能与主行动按钮竞争视觉重量。

## 5. 分步表单

表单由 `ValuationForm.vue` 管理，继续使用现有表单对象、默认值、数据列表和最终 payload。步骤状态只存在于前端组件内，不改变 API 合约。

| 步骤 | 用户任务 | 字段 |
| --- | --- | --- |
| 1 | 确认车辆身份 | 品牌、车型、车辆类型 |
| 2 | 描述使用情况 | 上牌年份、上牌月份、行驶里程、城市、车主次数 |
| 3 | 选择车辆配置 | 变速箱、燃油类型、发动机排量、座位数、颜色、排放标准 |
| 4 | 确认车况并提交 | 事故历史、全部输入摘要、开始估值 |

交互规则：

- 顶部显示“第 n 步 / 4”和线性进度，不使用装饰性动画。
- 继续按钮只校验当前步骤；错误信息紧邻对应字段，并将焦点移动到第一个错误字段。
- 返回按钮保留当前步骤数据；切换步骤不会触发 API 请求。
- 第 4 步显示可编辑的输入摘要，每项有明确的修改入口。
- 事故历史的 `unknown` 必须继续说明“unknown 不代表没有事故”，避免用户误解。
- 最终提交时保留现有 loading、错误和质量门禁状态。
- 所有输入和按钮维持至少 44px 的可操作高度；移动端使用单列布局。

## 6. 结果与证据

### 结果优先

`EstimatePanel.vue` 的空态、加载态和结果态继续存在，但结果态的首屏顺序调整为：

1. 参考价格。
2. 参考价格区间。
3. 质量门禁或实验状态。
4. 本次输入摘要。
5. “查看估值依据”入口。

价格区间仍然使用当前后端返回值，不新增未经验证的置信度计算。质量门禁失败时要在价格附近明确提示“实验结果”，不能只使用颜色区分。

### 依据展开

“查看估值依据”展开后显示现有 `ModelEvidence` 能力：模型版本、R²、MAE、数据来源、数据切分、特征契约、误差对比图和限制说明。默认折叠，展开后可收起；数据缺失时显示明确空态，不用 0 代替缺失值。

研究总览可以从顶部“模型说明”进入，保留排行榜、误差分组和来源边界等研究内容。估值结果页只展示与当前结果直接相关的摘要，避免复制整套研究面板。

## 7. 组件与数据流

预期调整边界：

- `App.vue`：默认视图改为估值；简化导航；保留全局 API 初始化、预测、历史和助手请求。
- `ValuationForm.vue`：加入步骤状态、分组渲染、逐步校验和第 4 步摘要；提交 payload 保持不变。
- `EstimatePanel.vue`：重排结果信息，增加依据展开入口或承载区域。
- `ModelEvidence.vue`：作为结果页的渐进展开内容复用，不修改证据计算逻辑。
- `main.css` / `base.css`：建立简洁的浅色层级、间距、响应式和焦点样式，减少侧栏、边框和阴影的视觉竞争。
- 现有 API、`researchEvidence.js` 和后端代码不改动。

数据流保持：

```text
表单本地状态 -> 当前步骤校验 -> 完整 payload -> /api/predict
                                              -> EstimatePanel
                                              -> /api/history 刷新记录
```

## 8. 状态与错误处理

- 初始化失败：保留全局 API 错误提示和重试按钮，不能显示一个看似可提交但无法工作的空表单。
- 步骤字段错误：只影响当前步骤，保留用户已填写内容。
- 预测中：禁用提交并显示明确的处理中状态，避免重复请求。
- 预测失败：在表单附近显示可读错误，结果区域保留上一次成功结果或明确空态，不能伪造价格。
- 质量门禁失败：价格仍可展示，但必须标识实验状态及限制。
- 证据缺失：展示字段级空态，不将缺失转换为 0 或“通过”。
- 无障碍：错误使用 `role="alert"` 或 `aria-live`，步骤变化更新可读标题，展开区域具有按钮语义和 `aria-expanded`。

## 9. 视觉与响应式约束

- 物理场景：用户在明亮桌面或手机环境中快速查询车辆价格，因此默认采用高可读的浅色画布和少量绿色行动色。
- 主体使用白色、深灰和单一绿色，不使用渐变、装饰性光晕或大面积玻璃效果。
- 主标题只承担当前任务，不在估值首页显示大段研究宣言。
- 使用留白、字号和分组区分层级，卡片只用于真正独立的输入或结果区域。
- 桌面端表单和结果可并列，但向导步骤只聚焦当前步骤；移动端全部单列。
- 遵循 `prefers-reduced-motion: reduce`，默认也不依赖动画才能理解流程。

## 10. 验证标准

- 现有前端测试继续通过，并补充分步表单的步骤切换、逐步校验、返回保留数据和最终 payload 测试。
- `npm run build` 通过。
- 在桌面和移动视口检查：标题、字段、按钮、错误信息、结果区间和展开证据不重叠、不溢出。
- 键盘可以完成四步表单，焦点顺序符合页面顺序，错误后焦点落在第一个错误字段。
- API 失败、预测失败、质量门禁失败和证据缺失均有可见且不误导的状态。
- 视觉验收以“用户是否能在第一眼开始估值、是否能在结果页理解价格性质”为准，而不是以研究字段数量为准。
