# 二手车估值模型质量门禁 v3 设计

## 1. 背景与目标

当前正式产物使用 2,059 条印度二手车公开数据训练 PyTorch MLP。测试集
`R2=0.8636`，但 10% 相对误差命中率仅为 `29.1%`，因此质量门禁为
`fail`。误差审计显示两个主要限制：价格从 4.9 万 INR 到 3,500 万 INR，
长尾明显；数据中有 1,050 个完整车型名称，其中 613 个只出现一次。
当前模型直接优化绝对价格 MSE，并且没有使用原始数据中的动力、扭矩、
驱动方式和车身尺寸字段。

本阶段的目标是建立一条无测试泄漏的表格模型竞赛流水线，优先使用
CatBoost 和对数价格目标提升相对误差表现，并在独立测试集上真实满足以下
质量门禁：

- `R2 > 0`；
- `acc_10 >= 0.50`；
- 模型 RMSE 低于使用开发集均值产生的测试集基线 RMSE。

不得通过降低阈值、删除合法豪华车样本、反复读取测试集调参或包装失败
结果来宣称达标。数据仍代表印度市场，价格单位仍为 INR。

## 2. 范围与非目标

本阶段包括数据契约 v3、特征工程、CatBoost/ExtraTrees/MLP 对照实验、
交叉验证模型选择、通用模型运行时、原子发布、模型证据 API 和前端证据区
增强。

本阶段不寻找或伪造中国市场数据，不转换为人民币，不扩大 LLM 助手能力，
不重做前端整体视觉，不将参考区间称为统计置信区间，也不提交原始 CSV、
密钥、虚拟环境或 `car-valuation-submit/`。

## 3. 总体架构

```text
公开原始 CSV
  -> 数据契约 v3 与字段解析
  -> 数据指纹、去重和外层测试集封存
  -> 开发集 5 折交叉验证
  -> CatBoost / ExtraTrees / MLP 模型竞赛
  -> 按交叉验证指标选择冠军
  -> 冠军在完整开发集重训
  -> 独立测试集一次性评估
  -> 质量门禁
  -> 原子发布模型、配置、排行榜、指标、误差报告和模型卡
  -> FastAPI 通用 ModelRuntime
  -> Research Console 模型证据区
```

训练代码不再假设正式模型一定是 `price_mlp.pt`。模型清单描述模型类型、
模型文件、目标变换、数据契约版本和特征版本，运行时按清单选择加载器。
现有 v2 MLP 产物继续受支持。

## 4. 数据契约 v3

### 4.1 新增字段

从 `car details v4.csv` 解析并加入以下字段：

- 类别字段：`seller_type`、`drivetrain`；
- 数值字段：`max_power_bhp`、`power_rpm`、`max_torque_nm`、
  `torque_rpm`、`length_mm`、`width_mm`、`height_mm`、
  `fuel_tank_liter`；
- 派生字段：`car_age`、`mileage_per_year`、`power_per_liter`、
  `footprint_m2`、`model_family`。

`model_family` 通过规范化完整车型文本并提取首个稳定车型词元生成，例如
`Amaze 1.2 VX ...` 映射为 `Amaze`。完整 `model` 仍保留给 CatBoost，
以同时利用具体配置和较稳定的车型族信息。

`vehicle_type` 在当前数据中恒为 `car`，`accident_history` 恒为 `unknown`。
二者继续存在于 API 契约和模型卡中，但不进入 v3 模型特征，且不得解释为
真实观测值。

### 4.2 解析和校验

功率与扭矩文本分别解析数值和转速，例如 `87 bhp @ 6000 rpm`、
`109 Nm @ 4500 rpm`。尺寸按毫米处理，油箱按升处理。无法解析的数值保留
为缺失值，不用零替代。类别缺失统一为 `unknown`。

价格必须为有限正数；年份、里程、排量、座位数和新增物理字段执行合理
范围校验。合法高价样本不因价格长尾被删除。完全重复记录在切分前删除。

产物记录源 URL、抓取时间、原始文件 SHA-256、适配器版本、清洗后样本数、
字段缺失率和目标分布。原始与标准化 CSV 继续由 `.gitignore` 排除。

## 5. 切分与防泄漏规则

数据按固定随机种子和 `log1p(price)` 分位区间分层。15% 作为外层测试集并
在模型选择期间封存，其余 85% 作为开发集。开发集执行确定性 5 折交叉
验证；每一折内的缺失值处理、编码、类别统计、早停和任何拟合步骤只能
访问该折训练部分。

切分产物记录开发集、测试集和 5 个折的原始行索引。各集合必须互斥且并集
覆盖清洗后数据。测试集只在冠军模型和训练配置冻结后评估一次。旧 v2 指标
作为历史基线展示，但所有候选模型必须在同一 v3 切分上重新训练，避免跨
切分直接排名。

## 6. 候选模型与目标变换

候选模型固定为以下三类：

1. CatBoostRegressor：原生处理高基数类别和缺失数值，使用 `log1p(price)`
   上的 RMSE、验证折早停和固定随机种子；
2. ExtraTreesRegressor：使用仅在训练折拟合的类别编码，作为强树模型基线；
3. PyTorch MLP：沿用现有预处理和网络边界，但使用 v3 特征与对数目标，
   作为深度学习对照。

每类模型使用代码中预先声明的小型参数集合，不进行无界搜索。CatBoost
最多 6 组、ExtraTrees 最多 4 组、MLP 最多 3 组。所有预测通过 `expm1`
还原到 INR，并拒绝非有限值；负值裁剪为零只作为数值防护并计入警告，
不能用于掩盖系统性失败。

## 7. 模型选择与最终评估

每个候选配置汇总 5 折的均值和标准差。冠军首先按平均 `acc_10` 排名；
若差距小于 0.01，则依次选择较低的 `median_ape`、较高的 `R2` 和较低的
运行复杂度。测试指标不进入冠军选择逻辑。

选定冠军后，在完整开发集上重训。CatBoost 的最终迭代数使用各折最佳迭代
数的中位数，不再访问测试集早停。最终报告至少包含：

- RMSE、MAE、R2、`acc_10`、`acc_20`、`median_ape`、RMSLE；
- 训练均值基线；
- 按测试集价格四分位的误差；
- 常见车型族与稀有车型族的误差；
- 已在开发集出现和未出现完整车型的误差。

质量门禁使用预先固定的三个条件。通过时才允许正式发布；失败时保存本地
实验报告但不覆盖当前正式产物，也不降低门禁。

## 8. 产物与原子发布

正式模型目录包含：

- `model_manifest.json`：模型类型、文件名、目标变换和契约版本；
- 冠军模型文件，例如 `price_model.cbm`、sklearn joblib 或 PyTorch checkpoint；
- `feature_config.json`；
- `metrics.json`；
- `leaderboard.json`；
- `error_analysis.json`；
- `model_card.json`；
- 需要时的 `preprocess.joblib`。

训练先写入同级临时目录。发布前必须重新加载临时模型，对固定样例执行推理，
校验所有 JSON 契约并执行质量门禁。全部通过后才以目录替换方式原子发布；
任一步骤失败时保留旧目录。正式模型文件需低于 GitHub 单文件 100 MB 限制。

## 9. 后端与前端集成

后端新增统一 `ModelRuntime`，为 CatBoost、sklearn 和旧 PyTorch 模型提供
相同的 `predict(frame)` 边界。`/api/predict` 保持现有请求和价格字段兼容，
并在响应中提供模型版本、冠军算法、数据契约版本和门禁状态。

`/api/model-card` 增加交叉验证排行榜、独立测试指标和分组误差摘要；
`/api/model-health` 仍只读取正式发布产物。历史记录继续保存货币和模型版本。

前端保持现有 Research Console 布局，只增强模型证据区，清楚区分开发集
交叉验证和独立测试结果，并展示冠军算法及价格分段表现。页面不得把交叉
验证结果描述为测试准确率。

## 10. 错误处理

以下情况必须显式失败，不得回退到手写估价规则：依赖缺失、模型文件损坏、
模型清单不完整、特征版本不匹配、输入无法满足契约、预测结果非有限值。
运行时模型故障返回 HTTP 503，用户输入不合法返回 HTTP 422。

下载或解析失败不能覆盖现有数据；训练或发布失败不能覆盖正式模型。失败
实验默认写入 Git 忽略目录，日志包含阶段、模型配置和根因，但不得包含密钥。

## 11. 测试策略

单元测试覆盖：

- 功率、转速、扭矩、尺寸和油箱解析，包括缺失、非法单位和小数；
- v3 数据列、常量字段排除和派生特征；
- 分层切分确定性、完整性和互斥性；
- 折内预处理与测试集隔离；
- `log1p/expm1` 往返和有限正数预测；
- 排行规则与测试指标不可参与冠军选择；
- CatBoost 新产物和 v2 MLP 产物加载；
- 门禁失败不发布、发布中断保留旧产物；
- API 元数据、历史兼容和 503/422 错误契约；
- 前端新增指标映射。

集成验收包括：从空数据目录下载并训练、重新加载正式产物完成预测、运行
全部后端与前端测试、构建前端生产包，并在桌面和移动视口完成浏览器冒烟
测试。

## 12. 验收标准

本阶段完成必须同时满足：

1. 公开数据可从记录的 URL 重建，数据和切分可复现；
2. 三类候选模型产生可审计的 5 折排行榜；
3. 独立测试集满足固定质量门禁，尤其是 `acc_10 >= 0.50`；
4. 后端测试、前端测试和生产构建全部通过；
5. 前后端运行时可加载冠军模型并返回带版本的 INR 预测；
6. Git 提交不包含数据、密钥、环境、失败实验或课设提交目录；
7. README 给出下载、训练、评估和启动命令，并明确印度市场限制。

若第 3 项未满足，本阶段不得标记为质量升级完成。应保留最优实验报告，
重新进入数据和误差分析，而不是修改验收阈值。
